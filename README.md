| `terraform-ibm-security-group` | [v2.0.0](https://github.com/terraform-ibm-modules/terraform-ibm-security-group/releases/tag/v2.0.0) | 18-09-2023 15:35 | <h3>Updates (<a href="https://github.com/terraform-ibm-modules/terraform-ibm-security-group/issues/131">#131</a>) (<a href="https://github.com/terraform-ibm-modules/terraform-ibm-security-group/commit/751e7bcc99cda627e8b81136947a00bae3e2fb8e">751e7bc</a>)</h3> <p>This release makes significant change in the way Security Group Rule resources are managed within Terraform state. In previous versions the rules were managed by an integer index (such as <code>ibm_is_security_group_rule.security_group_rule[0]</code>). This can make changing your rules array more difficult, since any reorder of the rules in your input variable can result in Terraform destroying rules and recreating them with a different index value.</p> <p>In order to make changes to the rules input more resilient, this release will switch to using the <code>name</code> property of each rule as the index for the Terraform state (<code>ibm_is_security_group_rule.security_group_rule["allow_https"]</code>). This allows Terraform to properly keep track of individual rules and changes without relying on their order in the input array, and should result in less rule destroy/create in the future.</p> <h3>BREAKING CHANGES</h3> <p>When this upgrade is applied to an existing set of deployed groups with rules, all rules will be <strong>DESTROYED</strong> and then recreated, which may involve a brief outage in network traffic for your deployed resources.
